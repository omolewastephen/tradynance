# Tradynance ‚Äî ERD (through Phase 2)

Source of truth is `web/prisma/schema.prisma`; this is a navigable summary, not a duplicate ‚Äî
update the schema first, then reflect changes here. The Prisma client is generated into
`packages/core/generated/prisma` (see `generator.output`) so the Next.js app and the
standalone `services/chain-watcher` share one client and one set of money-movement rules.

## Entities

**User** ‚Äî identity + role (`Role` enum covers admin roles from CLAUDE.md's spec: SUPER_ADMIN,
ADMIN, FINANCE, SUPPORT, COMPLIANCE, MODERATOR, DEVELOPER, AUDITOR, plus USER) + status +
KYC status + self-referential referral tree, plus better-auth-managed fields (`username`,
`displayUsername`, `twoFactorEnabled`) and `antiPhishingCode`. `referralCode` (own code) and
`referredById` are set server-side in a `user.create` databaseHook, not client input.

**Session** / **Account** / **Verification** / **TwoFactor** ‚Äî generated by `@better-auth/cli`
against this schema (Phase 1). Session is ephemeral/revocable ‚Äî see **LoginHistory** below for
the durable record. Do not hand-edit these models; regenerate via the CLI if better-auth's
config changes and reconcile the diff manually.

**LoginHistory** ‚Äî append-only, written via a `session.create` databaseHook in `src/lib/auth.ts`
on every sign-in (including 2FA-completed and email-verification auto-signins). Distinct from
`Session`: survives sign-out and session expiry cleanup, so it's the actual "login history" a
user sees in settings, not a live-session list.

**Asset** ‚Üí **AssetNetwork** ‚Äî a coin (BTC, ETH, USDT...) and the chains it can move on, each
with its own min-deposit/withdrawal-fee/memo-requirement.

**Wallet** ‚Äî one row per (user, asset, network). `balance` / `lockedBalance` are **cached**,
derived from `LedgerEntry` ‚Äî never mutated directly (see money-correctness invariant in the
schema header and CLAUDE.md hard convention #3). `derivationIndex` is the monotonic HD-tree
index this wallet's `depositAddress` was derived at (null for networks without live derivation).

**DerivationCounter** ‚Äî one row per network, source of the monotonic `derivationIndex`.
Incremented in the same transaction that provisions a wallet, so two wallets can never share
an index (hence never share a deposit address). See `packages/core/src/wallet/provision.ts`.

**LedgerEntry** ‚Äî append-only. Every credit/debit (deposit, withdrawal, trade fill, fee,
transfer, conversion, manual adjustment) is one row here, signed amount + balance snapshot.
No update/delete from application code; corrections are new entries. Deposits are written
here only by `creditDeposit` (`packages/core/src/ledger.ts`) ‚Äî the one idempotent entry point
shared by the chain-watcher and the admin manual-credit fallback.

**Deposit** ‚Äî CHAIN (chain-watcher) or MANUAL (admin fallback). `(network, txHash,
txOutputIndex)` unique ‚Äî the idempotency key stopping a re-scan from double-crediting. On
credit, `creditDeposit` writes one positive `LedgerEntry`.

**Withdrawal** ‚Äî the debit side, driven by `packages/core/src/withdrawal.ts`. State machine:
`AWAITING_CONFIRMATION` (created; funds NOT locked) ‚Üí confirm (email OTP + TOTP/password) ‚Üí
`PENDING` (funds locked via `Wallet.lockedBalance`, no ledger entry yet) ‚Üí admin approve ‚Üí
`COMPLETED` (one negative `LedgerEntry` for amount+fee; balance and lock both drop) OR reject
‚Üí `REJECTED` (lock released, no ledger entry). `confirmationCodeHash`/`confirmationExpiresAt`
gate the confirm step (single-use, short-lived). Available balance everywhere = balance ‚àí
lockedBalance.

**WithdrawalWhitelist** ‚Äî per-user trusted destination address book, network-scoped. With
`User.withdrawalWhitelistOnly`, withdrawals may only target these.

**Market** ‚Äî a trading pair (base/quote asset), precision + fee config. `dataSourceSymbol`
maps it to the upstream market-data feed.

**Ticker** ‚Äî one row per market, the latest live price + 24h high/low/volume/change, upserted
by `services/market-data` (polling `data-api.binance.vision`). A cache; not money-relevant.

**Watchlist** ‚Äî per-user favourited markets (unique on user+market).

**Order** ‚Üí **Trade** ‚Äî spot order book. An `Order` is user intent (side, type, TIF, price,
quantity, fill progress, avgFillPrice); a `Trade` is one match between a buy and a sell order
at a price/quantity, per-side fees, and `takerSide`. Placement + settlement is transactional
via `packages/core/src/trading-engine.ts` (`placeOrder`/`cancelOrder`) at Serializable
isolation ‚Äî fills write TRADE_FILL + FEE ledger entries for both sides; a GTC-limit remainder
rests and locks funds in the per-(user,asset) **SPOT wallet** (network="SPOT"). The
`services/market-maker` system user quotes resting liquidity so the book is fillable.

**AuditLog** ‚Äî append-only action log (actor, action, entity, metadata, IP). Nothing here is
ever deletable, per CLAUDE.md admin requirements.
